This repo contains three independent automation pipelines, all running on GitHub Actions and sending notifications via Telegram. They share Telegram secrets but are otherwise fully separate.

## Project 1: Soil Moisture Monitor

**Goal:** Daily soil moisture monitoring and irrigation recommendations for a garden in Utrecht Oost (sandy soil with clay component).

### Files
- `soil_model.py` ‚Äî FAO-56 Penman-Monteith ET0 model + data fetching
- `check_and_notify.py` ‚Äî daily runner + Telegram notifications
- `.github/workflows/daily-check.yml` ‚Äî cron at 06:00 UTC
- `docs/index.html` ‚Äî dashboard (vanilla HTML + Chart.js 4.4 + chartjs-plugin-annotation)
- `docs/data.json` ‚Äî **generated by the action, never edit manually**

### Daily flow
1. Fetch Weather Underground PWS history (30 days)
2. Fetch Open-Meteo (past 30 days + 7-day forecast)
3. Merge: WU wins for precip/temp, Open-Meteo supplies solar radiation (Rs)
4. Fetch `irrigations.json` from GitHub Gist
5. Run FAO-56 ET0 per day
6. Run soil water balance per zone (lawn + shrubs)
7. Write `docs/data.json` ‚Üí commit ‚Üí GitHub Pages deploy
8. If priority medium/high ‚Üí send Telegram notification

### Scientific model (do not change without explicit ask)
- ET0: FAO-56 Penman-Monteith using Tmax, Tmin, RHmean, u2, Rs, elevation, lat, doy
- **Dual Kc (FAO-56 ch. 7):** ETc = E + T, met
  - T = Kcb √ó Ks √ó temp_factor √ó ET0 (transpiratie uit wortelzone)
  - E = Ke √ó ET0 (directe bodemverdamping uit oppervlaktelaag), Ke = min(Kr ¬∑ (Kc_max ‚àí Kcb), few ¬∑ Kc_max) (Eq. 71), Kr lineair tussen REW en TEW (Eq. 74)
  - Effectieve Kc = (E + T)/ET0
- temp_factor = 0 below 5¬∞C, linear 0‚Üí1 between 5‚Äì8¬∞C, op 5-daagse rolling Tmean (lucht-Tmean als proxy voor bodemtemperatuur, gedempt)
- Kcb: seasonal via linear interpolation over monthly anchor points (jan‚Üídec)
- Twee buckets: diepe wortelzone (`water`) en oppervlaktelaag (`De`, depletie van 0..TEW)
- Regen/irrigatie vult oppervlaktelaag eerst tot TEW; overschot infiltreert naar wortelzone
- Drainage = excess wortelzone-water boven veldcapaciteit (instantaan, dezelfde dag)
- Ks stress factor: linear decrease when depletion > p √ó AWC (FAO-56 Eq. 84)
- p (depletion fraction): 0.40 voor gras, 0.50 voor struiken (FAO-56 Tabel 22)
- Interceptie: canopy-saturation curve `I = C ¬∑ (1 ‚àí exp(‚àíP / C))` met C = 1.0 mm (gras) / 1.5 mm (struiken). Smooth: kleine events worden grotendeels onderschept, grote events verzadigen de canopy bij C.
- Oppervlaktelaag-parameters: TEW 18 mm, REW 8 mm, Ze 0.10 m (FAO-56 Tabel 19, sandy-loam ‚Üí klei-versterkte zand)
- few = min(1 ‚àí fc, fw), met fc = 0.95 (gras) / 0.75 (struiken ‚Äî canopy + mulch/bladafval), fw = 1.0 (regen of sproeier) / 0.30 (druppelirrigatie struiken)
- Kc_max = 1.20 (sub-humide klimaat, u2 ‚âà 2 m/s, FAO-56 Eq. 72)
- Soil params: FC 0.20, WP 0.09 v/v ‚Äî clay-amended sand (ophoogzand) calibration for Utrecht Oost
- Zones: lawn (Zr 0.20m), shrubs (Zr 0.42m) ‚Äî each with own Kcb curve
- Irrigation rates: sprinkler 20mm/hr (lawn), drip 2mm/hr (shrubs)
- Wind: anemometers (Open-Meteo + WU PWS) zijn op 10m; FAO-56 Eq. 47 log-law correctie naar 2m wordt toegepast (u2 = u10 √ó ~0.748)
- **WU stralingsbiascorrectie (`wu_bias.py`):** het WU-station heeft een radiatieve warm-bias die lineair met de instraling meeschaalt (gediagnosticeerd in Project 7). Op WU-gemeten, niet-forecast dagen wordt `Tmax` gecorrigeerd met `T ‚àí SOLAR_BIAS_SLOPE ¬∑ max(0, instraling_W/m¬≤)` v√≥√≥r ET0; driver = de eigen WU-pyranometer (`solarRadiationHigh`, dagpiek ‚Äî valt op Tmax-piek), met de Open-Meteo dagpiek als fallback. `Tmean` wordt gecorrigeerd met de Open-Meteo 24u-mean (laag-risico: enkel de koude `temp_factor`); `Tmin` (nacht) blijft ongemoeid. Rauwe `Tmax/Tmin/Tmean` blijven behouden; correcties zijn additief (`Tmax_corr`, `Tmean_corr`, `bias_corr`, `bias_solar_src`). Forecast/pre-WU/archief-dagen blijven ongecorrigeerd. `SOLAR_BIAS_SLOPE` wordt gekalibreerd door Project 7 (zie daar).
- **State carry-over:** `theta_end` per zone wordt elke run weggeschreven in data.json. De volgende run leest dit als `seed_theta` zodat het 35-daagse warmup-venster niet elke run vanaf 30%-uitputting hoeft te starten ‚Äî convergeert naar consistente initi√´le conditie.

## Project 2: Personal Weather Briefing

**Goal:** Daily personal weather briefing via Telegram, structured around fixed time blocks relevant to daily routine.

### Files
- `weather_briefing.py` ‚Äî fetch Open-Meteo + build Telegram message
- `.github/workflows/weather-briefing.yml` ‚Äî cron at 01:00 UTC

### Logic
- Fetches Open-Meteo hourly forecast for Utrecht
- Parses 24h window: temp, apparent_temp, precip, pop, uv_index
- Builds message per time block (configured at top of script):
  - Two block sets: `WEEKDAY_BLOCKS` (Mon‚ÄìThu) and `PEUTER_BLOCKS` (Fri‚ÄìSun)
  - Weekday blocks:
    - Fietstocht 06:00‚Äì06:30 (Tuesday + Thursday only, weekdays={1,3})
    - KDV brengen 08:00‚Äì09:00 (Mon/Tue/Thu, weekdays={0,1,3})
    - Naar kantoor 08:00‚Äì09:00 (Wednesday only, weekdays={2}) ‚Äî cycle to work without Ted
    - Naar huis 16:30‚Äì17:30 (Monday‚ÄìThursday, weekdays={0,1,2,3})
    - Sport vrouw 19:00‚Äì20:00 (Monday + Wednesday only, weekdays={0,2})
  - Peuter blocks (Friday + Saturday + Sunday):
    - Na fruit 09:30‚Äì11:30
    - Na dutje 15:00‚Äì17:00
- UV windows via linear interpolation between hourly values, rounded to 30 min
- Windows shorter than 15 min are filtered out
- HOME_COORDS + HOME_RADIUS_KM (10km from Utrecht) determines home vs vacation mode:
  - Home: full output with time blocks + UV
  - Vacation: UV info only, blocks skipped
- UV thresholds: UV_MODERATE = 3.0, UV_HIGH = 5.0
- `DRY_RUN=1` env var: print output without sending to Telegram (for local testing)

### UV cloud correction
Open-Meteo's `uv_index` field is only weakly cloud-corrected and reports clear-sky-like values on overcast/rainy days. We therefore fetch `uv_index_clear_sky` and apply the Josefsson & Landelius (2000) cloud modification factor `CMF = 1 - 0.75 * (cloud_fraction)^3.4` using the hourly `cloud_cover` from the same call. The corrected value is what `uv_windows()` thresholds against. If `uv_index_clear_sky` is missing for a given hour, fall back to the raw `uv_index`.

## Project 5: Grasmaai-adviseur (Mowing Advisor)

**Goal:** Tell when the lawn is due for a mow again, and recommend the cutting height (30/40/50mm) based on the weather ‚Äî biased toward a strong root system.

### Files
- `mowing_advisor.py` ‚Äî growth model + Telegram + writes `docs/mowing_data.json`
- `.github/workflows/mowing-notify.yml` ‚Äî cron at 07:00 Amsterdam + manual dispatch
- `docs/mowing.html` ‚Äî dashboard (vanilla HTML + Chart.js 4.4 + annotation), separate from the soil dashboard but reuses the same Gist + `localStorage` token (`gist_id`, `gh_token`)
- `docs/mowing_data.json` ‚Äî **generated by the action, never edit manually**
- `mowing_state.json` ‚Äî tiny notification bookkeeping (committed by the action), like `sandbox_state.json`

### Daily flow
1. Read the mow log from the **same Gist** as the soil project, file `mowings.json` (shape `{"YYYY-MM-DD": {"length_mm": 40}}`, length ‚àà {30,40,50}). Browser writes it via the "‚úÇÔ∏è Ik heb gemaaid" modal; Python reads it **read-only**.
2. Read `docs/data.json` (the soil project's published artefact) **read-only** for the growth driver.
3. Accumulate grass growth since the last mow, decide ready/optimal day + cutting height.
4. Write `docs/mowing_data.json` ‚Üí commit; send Telegram only when there's something to say.

### Growth model (the brain)
- Daily growth unit `GU = lawn_T √ó heat_derate(Tmax)`, where `lawn_T` (actual transpiration from `data.json`) already encodes season, cold, drought and demand via FAO-56. The heat derate (1.0 ‚â§24¬∞C ‚Üí `HEAT_FLOOR` 0.25 ‚â•35¬∞C) reflects that cool-season turf keeps transpiring in heat but stops elongating. Cold/water stress are **not** re-applied (already inside `lawn_T`).
- Accumulate GU since the last mow (mow day resets the accumulator). **Ready** when `accum ‚â• READY_GU_effective` and not dormant.
- `READY_GU` default 14.0; **self-calibrates** to the median accumulated-growth-between-logged-mows once ‚â•4 intervals exist (clamped 8‚Äì24).
- **Dormancy guard:** 7-day mean GU < 0.4 ‚Üí winter, suppress all nudges.
- **Cutting height:** 50mm when heat/drought is coming (‚â•2 days ‚â•27¬∞C in next 5, or `lawn_depletion ‚â• 55%`) to protect roots; 30mm only when cool (<22¬∞C), moist (<35% depletion), and in growth months (Apr‚ÄìOct); 40mm otherwise (default).
- **Fallback:** if `data.json` is missing/stale (>36h), switch to GDD-only mode (own Open-Meteo Tmean fetch, base 6¬∞C) and flag it in the message/dashboard.

### Notification cadence
Tiny `mowing_state.json` (`last_notified_date`, `last_notified_kind`, `last_seen_mow_date`). One message/day max; re-nudge after 3 unmowed days; silent when not ready, dormant, or on a cold start (no real mow logged yet). A newly logged mow resets the notification memory.

### Relation to other projects
Independent except for **one deliberate read-only consumption** of `docs/data.json` (never imports `soil_model`, never writes `data.json`). Scheduled at 07:00, one hour after the 06:00 soil run. No new secrets ‚Äî reuses Telegram + `GIST_ID`/`GIST_TOKEN`.

## Project 6: Tado Window Advisor (raam-koeladvies)

**Goal:** On warm summer days, cool the house by telling ‚Äî per room ‚Äî when to open windows (outside cooler than the room) and when to close them (heat-in). Ventilation handles air quality; this is purely a thermal decision. Advice only ‚Äî no window is actuated.

### Files
- `window_advisor.py` ‚Äî hourly brain: tado auth ‚Üí per-room temps ‚Üí decide ‚Üí Telegram + writes `docs/window_data.json`
- `tado_auth_bootstrap.py` ‚Äî **one-time** local device-code authorization, seeds the refresh token into the secret Gist
- `.github/workflows/window-notify.yml` ‚Äî kwartiercadans via een self-driven loop (cron-kicks elke ~20 min, daarna 20 iteraties op `:00/:15/:30/:45` mark per run, ~5u wall time) + manual dispatch (`dry_run` input), `permissions: contents: write` (commits the dashboard JSON)
- `docs/window.html` ‚Äî dashboard (vanilla HTML + Chart.js 4.4 + annotation), view-only (no Gist token needed); cross-linked with the soil + mowing dashboards
- `docs/window_data.json` ‚Äî **generated by the action, never edit manually**

### Rooms in scope
tado zone names, matched case-insensitively: **Living room, Ted, hotties, office** (edit `ROOMS` at the top of `window_advisor.py`).

### Hourly flow
1. Read the rotating tado refresh token from the **secret Gist** (`tado_token.json`), exchange it for an access token, and **immediately write the rotated refresh token back** (tado rotates on every refresh).
2. tado API: `/me` ‚Üí home id; `/homes/{id}/zones`; per matching zone `/state` ‚Üí inside temperature + humidity.
3. Outside temp **now** from WU PWS current obs (`metric.temp`), fallback Open-Meteo current hour. On a WU reading the **stralingsbiascorrectie** (`wu_bias.py`) is applied first ‚Äî driver = WU `solarRadiation` now, Open-Meteo `shortwave_radiation` as fallback ‚Äî so `outside_now` (and therefore the microklimaat-bias-blend, `decide()`, and `outside_history.temp`) reflect the corrected temperature. The Open-Meteo fallback path is left uncorrected (already a model value).
4. Open-Meteo hourly forecast (2 days) ‚Üí day-max gate + "open again ~HH:00" lookahead. Forecast timestamps are made timezone-aware (Open-Meteo returns na√Øve local times) so they compare safely with `datetime.now(TZ)`.
5. Per-room decision with hysteresis, update per-room state in the Gist (`window_state.json`).
6. Telegram only the rooms whose advice **flipped** this run ‚Üí weerbriefing-groep.
7. **Always** write `docs/window_data.json` (even on suppressed/cool days) and commit it ‚Äî this is the dashboard artefact. Telegram cadence is unchanged.

### Dashboard prediction (`docs/window_data.json` ‚Üí `docs/window.html`)
The dashboard predicts, per room, *when* the window can be opened today (or "keep closed today") by combining the forecast with the station and the room-temperature trend ‚Äî all computed in `window_advisor.py`:
- **Station bias correction (smart blend):** `bias = WU_now ‚àí Open-Meteo_now` is the local microclimate/calibration offset. It is added to the future forecast and **decays linearly to zero over `BIAS_DECAY_H` (12h)** ‚Äî near-term hours are anchored to the station, far-term relax to the raw model. WU unavailable ‚Üí `bias = 0`.
- **Room-temp trend:** a rolling per-room inside-temp history (and outside-now) is kept in `window_data.json`; each run appends the current sample and trims to `HISTORY_KEEP` (~48). The trend `slope` (¬∞C/h, least-squares over `TREND_WINDOW_H` hours, clamped to ¬±`TREND_MAX_SLOPE`) is projected forward but **damped** (`min(hours, TREND_CAP_H)` then flat) ‚Äî a heuristic, **not** a thermal house model.
- **Crossover:** the first future hour (‚â§ `PREDICT_HORIZON_H`) where `inside_proj > COMFORT_HIGH` **and** `out_corr ‚â§ inside_proj ‚àí OPEN_MARGIN` is the predicted open time; it closes again when `out_corr ‚â• inside_proj ‚àí CLOSE_MARGIN` ‚Üí `open_intervals`. No crossover today ‚Üí "vandaag dicht houden".
- **Ventilatie-RH (schimmel):** per room the outside RH is converted to the room's temperature via `convert_rh()` (`vent_rh`) ‚Äî the RH the room would approach if ventilated with outside air, since absolute vapour pressure is conserved when air changes temperature (Magnus/Tetens `_es`, FAO-56 Eq. 11). Computed from the **raw** WU temp + its own RH reading (a consistent sensor pair, *not* the radiation-bias-corrected temp), Open-Meteo `relative_humidity_2m` + `temperature_2m` as fallback. The dashboard shows it behind the indoor humidity as `(X% buiten)`, tinted green when below indoor RH (ventilating dries ‚Üí less mould risk) and clay when above (ventilating adds moisture, e.g. a warm humid summer day). **`vent_rh` is no longer purely informational ‚Äî it now also feeds `decide()` (see "Humidity-balanced decision" below).**
- Sparklines clamp to a minimum span (~3.5¬∞C) and draw a faint `COMFORT_HIGH` reference line, so a stable room reads flat instead of magnifying measurement noise.
- `outside_history` also records the raw Open-Meteo value of the same hour (`om`) next to the used `temp`, so the chart can look *backwards* and show **weerstation (gemeten) vs. Open-Meteo (ruw model)** ‚Äî making station/model divergence (e.g. a station reading too warm on a sunny evening) visible. The `om` field is additive; older samples without it just leave a gap in the model line until history accumulates.
- Dashboard panels: station-vs-model bias readout + warm/cool gate (with its own outside trend arrow + sparkline, mirroring the rooms), per-room cards (inside temp, open/dicht stamp, trend arrow + sparkline, humidity, status line), a temperature chart with three outside series (station-measured + used/calibrated + raw Open-Meteo model, past and future) plus per-room inside projections, `COMFORT_HIGH`/`WARM_DAY_MAX` lines, a "nu" marker, and a day-aligned x-axis (date labels at midnight, hour labels at 06/12/18u); a per-room open-window timeline; and a **temperature √ó humidity scatter** (`#th-chart`) where each room (and `buiten`, plotted from its measured RH) is one dot at its current `(inside, humidity)` with a single angled trend arrow (`dx` = temp trend, `dy` = humidity trend, projected ~2h) showing where it's heading, plus `RH_COMFORT`/`RH_HARD_CAP` reference lines ‚Äî the visual of the humidity-balanced decision.
- **Humidity trend (scatter `dy`):** the rolling history additively records `hum` (indoor RH per room, measured outside RH in `outside_history`); `room_trend(history, now, key="hum", clamp=RH_TREND_MAX)` reuses the temp least-squares slope to give `hum_trend`/`outside_hum_trend` (%RH/h). Like `om`, older samples lack `hum`, so arrows are temp-only until history fills.

### Decision logic (tunable constants at top of script)
- `OPEN_MARGIN 1.5`, `CLOSE_MARGIN 0.5`, `WARM_DAY_MAX 22.0`.
- **Per-room comfort band** `ROOM_COMFORT = {room: (low, high)}` ‚Äî Living room `19.5‚Äì22.0`, Ted `17.0‚Äì18.0`, hotties `16.0‚Äì18.0`, office `20.0‚Äì22.0`. `high` is the cool/open trigger, `low` the stop-overcooling/close trigger. `COMFORT_HIGH 23.5` is only the fallback (`low = high = COMFORT_HIGH`) for any room **not** in `ROOM_COMFORT` ‚Üí reduces to the old single-threshold behaviour.
- **OPEN** when `inside > high + humidity_offset(vent_rh)` and `outside ‚â§ inside ‚àí OPEN_MARGIN` (the muggy/dry humidity shift is described below).
- **CLOSE** when `outside ‚â• inside ‚àí CLOSE_MARGIN` (heat-in), or `inside ‚â§ low` (cool enough ‚Äî don't overcool) **unless banking cooling** (see below).
- In-between (`low < inside ‚â§ high`) ‚Üí keep current advice (dead-band ‚Üí no flapping).
- **Humidity-balanced decision (humid house, dislikes warmth):** ventilating to cool can also drag in muggy air, so `vent_rh` (the projected room RH after ventilating, see "Ventilatie-RH" above) now shifts the open decision. The whole thermal logic is unchanged on normal dry days; only the open trigger and a veto are touched. Constants at the top of `window_advisor.py`:
  - `humidity_offset(vent_rh)` = `RH_TEMP_K ¬∑ (vent_rh ‚àí RH_COMFORT)`, clamped to `[‚àíRH_BONUS_MAX, +RH_PENALTY_MAX]`. **Muggy** (`vent_rh > RH_COMFORT 60%`) **raises** the open threshold (the room must get hotter before a window opens); **dry** outside air gives a small bonus that lowers it. `RH_TEMP_K 0.15` spans nearly the full 0‚Äì2¬∞C penalty across the 60‚Üí72% band (target‚Üíveto); the bonus is deliberately small (`RH_BONUS_MAX 0.5` vs `RH_PENALTY_MAX 2.0`) so ordinary days barely change.
  - **Hard veto (`RH_HARD_CAP 72%`):** if ventilating would push the room past this projected RH, `decide()` returns `dicht` ‚Äî never ventilate into that-muggy air, and close an open window if outside turns that muggy (e.g. a humid downpour).
  - **Dehumidify trigger (`open_desire`):** a muffe-but-not-warm room (indoor `humidity ‚â• RH_DRYOUT_MIN 65%`) **opens** when the outside air is clearly drier (`vent_rh ‚â§ humidity ‚àí RH_DRYOUT_MARGIN 8%`), there's no heat-in (`outside ‚â§ inside`), and it won't overcool (`inside > low`). This is the only way humidity can *create* an open; otherwise it only discourages/vetoes one.
  - `open_desire()` is the single source of truth for both `decide()` and the dashboard `open_now`, so advice and dashboard can't drift. The forward open-time predictor shifts its threshold by the **current** `humidity_offset` (no per-hour RH forecast ‚Äî a known static approximation); the dehumidify trigger isn't projected forward. Per-room `rh_offset`/`rh_veto`/`dryout` are written to `window_data.json` and surfaced as a humidity chip on each card; Telegram adds `‚Äî ontvochtigen (buiten droger)` / `‚Äî buiten te muf (RH ~X%)` suffixes when humidity is the reason for a flip.
- **Smoothed decision temp (`SMOOTH_WINDOW_H 0.75`):** before `decide()`/`open_now`, the (already bias-corrected) outside temp is replaced by the **median** over the last `SMOOTH_WINDOW_H` of `outside_history` (incl. the current sample). Quarter-hourly readings can swing >2¬∞C ‚Äî more than the 1.0¬∞C dead-band ‚Äî when a short, heavy rain shower briefly drops the outside temp by evaporative cooling and then recovers; the median ignores these lone dips so a passing shower no longer flips the advice (and you don't want a window open during a downpour anyway). The **raw** corrected reading still drives `outside_now`, `bias` and `outside_history.temp` (history is never the smoothed value, so the median can't compound); `outside_smoothed` is the additive readout of the value `decide()` actually saw, and Telegram shows it too so the printed ¬∞C justifies the advice.
- **Banking cooling (`warm_ahead`):** if the max temp in the next 24h ‚â• `WARM_DAY_MAX`, a room is **not** closed just because it dipped below its `low` ‚Äî windows stay open through the night to keep banking coolness for the warm day ahead, as long as outside stays cooler. Heat-in still closes. On a cool day ahead the old "cool enough ‚Üí close" behaviour applies. `warm_ahead` is a forward-looking 24h check (not calendar-day) so it's correct deep at night.
- **Cooling-only gate:** cool forecast day (max `< WARM_DAY_MAX`) and no warm room ‚Üí no advice.
- **Reopen hint** (`‚Äî buiten zakt rond HH weer onder binnen`) on a "Sluit" message only shows when outside is currently *above* the reopen threshold (`outside > inside ‚àí OPEN_MARGIN`), i.e. a genuine heat-in close. If outside is already below it, the hint would be meaningless and is suppressed.
- **Brief-heat-in hold (`MIN_CLOSE_H 1.0h`):** an *open* room is **not** closed for a heat-in moment that the forecast says is already over within `MIN_CLOSE_H` ‚Äî if `next_reopen()` (first forecast hour where `out_raw ‚â§ inside ‚àí OPEN_MARGIN`) lands within the window, `decide()` holds the previous advice instead of flipping to `dicht`. Not worth shutting a window for ~15 min only to reopen. Only applies when the window is currently open (a closed room stays closed) and the muf hard-veto still takes precedence. Threshold is forecast-hour-granular (Open-Meteo is hourly), so it effectively means "the very next forecast step already shows it cool enough ‚Üí ignore the dip".

### tado auth (the catch)
Since March 2025 tado uses the OAuth2 **device-code flow** (no username/password). Refresh tokens **rotate** (each refresh revokes the previous) and live ‚â§30 days; hourly runs keep the chain alive. The refresh token lives **only** in the secret Gist + Actions secrets ‚Äî **never committed** (public repo). Public app `client_id` is hardcoded (not a secret). If the chain breaks, re-run `tado_auth_bootstrap.py`.

### Notification cadence

One check per 15 minutes, message only on a change. `DRY_RUN=1` prints instead of sending (token + state are still persisted ‚Äî rotation must not be skipped). `fetch_open_meteo` retried 3√ó met korte backoff (geen retry ‚Üí ~17% iteratieverlies bij incidentele 5xx-hiccups gemeten).

Independent. **Commits `docs/window_data.json` each run** (workflow is now `contents: write`) so the dashboard has fresh data ‚Äî this replaces the earlier "no commit-back" design. The tado **refresh token and per-room advice state still live only in the secret Gist** (`tado_token.json`, `window_state.json`) and are **never committed**; only the non-secret dashboard JSON is published. Reuses Telegram (group), `GIST_TOKEN`, and the WU secrets.

## Project 7: Weerstation-nauwkeurigheid (Station Accuracy Diagnostic)

**Goal:** Quantify *whether* and *when* the WU PWS reads inaccurately ‚Äî especially the suspected warm bias on sunny (late-)afternoons ‚Äî by comparing its hourly temperature against Open-Meteo ERA5 and stratifying the deviation.

### Files
- `station_accuracy.py` ‚Äî fetch WU hourly history + Open-Meteo ERA5 archive, pair by hour, stratify the bias, write `docs/accuracy_data.json` + print a report (optional Telegram summary)
- `.github/workflows/station-accuracy.yml` ‚Äî **manual-dispatch only** (no cron), inputs `days` + `notify`, `permissions: contents: write` (commits the dashboard JSON)
- `docs/accuracy.html` ‚Äî dashboard (vanilla HTML + Chart.js 4.4 + annotation), cross-linked with the other dashboards
- `docs/accuracy_data.json` ‚Äî **generated by the action, never edit manually**

### Method
- **Reference = Open-Meteo ERA5** (archive, ~5-day lag). A grid-scale model, *not* ground truth, so a raw `WU ‚àí model` gap blends (a) sensor error and (b) real microclimate. They're separated by *behaviour*: a radiative/siting fault scales with solar radiation, worsens at low wind, and (for radiative) vanishes at night; a constant offset is equal day and night.
- Pulls WU **hourly** history (`v2/pws/history/hourly`, per-day calls) ‚Äî distinct from the soil project's daily endpoint.
- Open-Meteo archive hourly: `temperature_2m`, `shortwave_radiation`, `cloud_cover`, `wind_speed_10m` (timezone=UTC for clean hour alignment).
- Pairs on the common UTC hour, `bias = WU ‚àí model`, then aggregates: diurnal curve (per local hour), bias vs cloud bins, bias vs solar bins, bias vs wind bins, the headline "sunny late-afternoon" subset (13‚Äì19u local, <25% cloud) vs the rest, and least-squares slopes (¬∞C per 100 W/m¬≤, ¬∞C per km/h).
- **Calibration output for `wu_bias.py`:** the bias is also fit against **two** radiation drivers ‚Äî Open-Meteo grid solar *and* the WU station's own pyranometer (`solarRadiationHigh`, co-located ‚Üí captures direct-sun/broken-cloud bursts the grid smooths away). The report compares their bias‚Üîsolar correlations and prints the recommended driver + `SOLAR_BIAS_SLOPE` (= the chosen `slope_per_100 / 100`). The co-located WU driver wins when its correlation is equal-or-tighter. This is the single source of truth for the constant in `wu_bias.py` ‚Äî re-run to recalibrate and paste the printed value. (`accuracy_data.json` gains additive `wu_solar_slope_per_100`, `solar_bias_corr`, `wu_solar_bias_corr`, `recommended_slope`.)

### Relation to other projects
Read-only with **one deliberate output**: it never imports or writes other projects' artefacts, but it is the **calibration source** for `wu_bias.py`'s `SOLAR_BIAS_SLOPE` (a hand-copied constant, no runtime coupling). Reuses the `WU_*` and Telegram secrets only. `WU_STATION_ID` is a secret and is **never** written to `accuracy_data.json` or logs.

## Project 8: Ventilatie-tweeling (Cross-Ventilation Digital Twin)

**Goal:** A self-calibrating *digital twin* of the house. It's the **inverse** of Project 6: there the model tells you which windows to open and you act; here **you tell the model which windows / vents / doors are open**, it **predicts each room's temperature** ("derived temperatures"), **shows the error** against the real tado temps, and **auto-tunes its own parameters** so it improves the longer it runs. A passive "what would cool best" suggestion is shown alongside but is advisory only. **No Telegram.**

It's a **grey-box multi-zone airflow network + 2-node RC thermal model with online calibration** (a miniature, self-calibrating cousin of EnergyPlus AirflowNetwork). The physics (sun position, orifice flow, mass balance) is exact; the inputs (wind pressure coefficients, opening geometry, internal-door state) are approximate and get learned ‚Äî a strong *relative* ranker, not a litres/sec oracle.

### Files
- `airflow_model.py` ‚Äî airflow network + 2-node thermal model + calibration loop + runner
- `house_model.json` ‚Äî committed **non-secret** house geometry (rooms, windows w/ fa√ßade orientation + size + how-far-open + height, vents, internal doors). Scaffolded template; edit the numbers to match the real house. **Never put secrets here.**
- `.github/workflows/airflow-notify.yml` ‚Äî self-driven quarter-hour loop (cron-kick `9,29,49` ‚Äî a few min after the window run so `window_data.json` is fresh), `permissions: contents: write`, concurrency group `airflow-advisor`
- `docs/airflow.html` ‚Äî dashboard: floor-plan with airflow arrows + wind/sun, predicted-vs-actual temperature chart, learning panel (RMSE-over-time + params), passive suggestion, and the **"Stel ramen/roosters in"** reporting modal
- `docs/airflow_data.json`, `docs/airflow_learned.json` ‚Äî **generated by the action, never edit manually** (dashboard artefact + persisted learned params/RMSE history)

### Per-run flow
1. Read `docs/window_data.json` **read-only** for the actual tado per-room inside temp + humidity (+ history) ‚Äî the ground truth to learn against. **No tado auth** (avoids the rotating-token conflict with Project 6).
2. Fetch its own Open-Meteo (past 3d + 2d forecast): temp, RH, precip, **wind speed + `wind_direction_10m`** + gusts, and `shortwave/direct/diffuse_radiation`.
3. Read the user-reported opening log (`house_openings.json`) **read-only** from the non-secret `GIST_ID` Gist; reconstruct what was open per 15-min step (`openings_at`).
4. Build a 15-min driver timeline (incl. per-window solar **through the glass** via `sun_position` NOAA + fa√ßade incidence).
5. `calibrate()` ‚Äî damped Gauss-Newton on the learnable scales (global `cp_shelter/cd/vent_eff`; per-room `c_air/c_mass/ua_env/solar_gain`), minimizing Œ£(predicted ‚àí tado)¬≤ over the window, then nudges `LEARN_RATE 0.5` toward the optimum (online ‚Üí converges over runs). Persists params + `rmse_history`.
6. `simulate()` the predicted temps with the learned params; `suggest()` the best cooling window-set (advisory).
7. Write + commit `docs/airflow_data.json` + `docs/airflow_learned.json`.

### Physics (do not casually retune the structure)
- **Airflow network** (`solve_network`): zone pressures solved by Newton so mass balances at each room/junction node. Orifice flow `Q = sign(ŒîP)¬∑Cd¬∑A_eff¬∑‚àö(2|ŒîP|/œÅ)` with a **laminar/turbulent two-regime** crossover at `DP_LAM 0.1 Pa` (the ‚àö law has infinite slope at ŒîP=0 ‚Üí ill-conditioned Jacobian; the linear regime below DP_LAM fixes convergence ‚Äî standard CONTAM/AIRNET). Wind pressure `Pe = shelter¬∑Cp(Œ∏)¬∑¬ΩœÅU¬≤`, `Cp(Œ∏)` a two-cosine low-rise fit (loef +0.7 ‚Üí side ‚àí0.4 ‚Üí lee ‚àí0.25). Stack term `‚àù œÅ¬∑g¬∑z¬∑ŒîT`. Per-room infiltration leak `LEAK_AREA` keeps the system well-posed.
- **2-node RC thermal** (`simulate`): per zone an air node `T_a` (fast) + wall/furniture mass node `T_m` (slow), integrated **implicitly** (backward Euler ‚Üí one linear solve per substep via `solve_linear`; explicit Euler blows up under strong door coupling). Heat terms: ventilation `œÅcp¬∑Q_fresh¬∑(T_out‚àíT_a)`, advective door coupling between rooms, envelope conduction, air‚Üîmass coupling, solar `Œ£ œÑ¬∑shading¬∑I_window¬∑glass_m2` split `f_air` air / `1‚àíf_air` mass, **party-wall coupling to heated neighbours** and **internal gains** (see below).
- **Tussenwoning-termen (jaren-'20 rijtjeshuis).** Without these the model only conducted each room to the *cold* outside and predicted every room ~1.3¬∞C too cold on windy days, driving every learnable param to its heat-retaining bound (the saturation was the tell of a missing term). Two terms fix it, both on the air node, both with a learnable per-room scale:
  - **Party walls ‚Üí neighbours** (`ua_party`): a near-constant warm boundary. Conductance `UA_party¬∑(NEIGHBOR_TEMP ‚àí T_a)` with `NEIGHBOR_TEMP 20¬∞C` fixed (no neighbour sensor) ‚Äî the woningscheidende muren grenzen aan verwarmde buren bij ~alle kamers, so each room is pulled toward ~20¬∞C instead of toward outside. Base ~ envelope-UA; a corner room with fewer neighbours learns it lower.
  - **Internal gains** (`q_int`): `volume¬∑INTERNAL_GAIN_WM3¬∑q_int¬∑profile(t)` W, where `internal_gain_profile` is a **day/night schedule** (awake 07‚Äì23u = 1.0, asleep = `INTERNAL_NIGHT_FRACTION 0.5` ‚Äî sleeping bodies + standby aren't zero). People/cooking/appliances.
  - The two are partly degenerate on flat data but separate on real day/night variation (party = proportional-to-deficit toward 20¬∞C, gains = scheduled constant). Validated: together they lift a cold steady-state by +1.4‚Ä¶+2.0¬∞C and free `c_air`/`c_mass` off their ceilings.
- **Reported openings** are graded (`closed`/`tilt`/`open` or 0..1; `tilt`‚Üí`tilt_frac`) and **timestamped** (a snapshot holds until the next), so the model knows what was open during each past interval.

### data schema (additive only ‚Äî never break existing fields)
`airflow_data.json`: `generated_at`, `as_of_local`, `source`, `weather` (outside temp/RH, wind speed/dir/gust, sun az/el), `openings`, `controls` (elements + current state, for the modal), `rooms.<id>` (`actual_temp`, `predicted_temp`, `predicted_mass_temp`, `error`, `ach`, `solar_w`, `party_w` (net W from neighbours), `internal_w` (internal gains W), `humidity`, `comfort_low/high`, `predicted_series[]`, `actual_series[]`, `params`), `flows[]` (per-opening m¬≥/s for the floor-plan), `suggestion` (`instructions[]`, `ranked[]`, `keep_closed`, `headline`), `learned` (`params`, `rmse`, `rmse_history`), `house_meta`.

### Reporting interaction (the inverse loop)
The dashboard "Stel ramen/roosters in" modal writes a timestamped snapshot to `house_openings.json` in the **non-secret `GIST_ID` Gist** (same browser `gist_id`+`gh_token` localStorage pattern as the soil/mowing modals), then `workflow_dispatch`es `airflow-notify.yml`. The browser is the only writer of the log; Python reads it read-only.

### Relation to other projects
Fully isolated: the **only** dependency is the read-only consumption of `docs/window_data.json` (it never imports `window_advisor`'s tado/main path ‚Äî only the pure helpers `convert_rh`, `ROOM_COMFORT`, `RH_HARD_CAP`), and **nothing else depends on Project 8**. Reuses `wu_bias.correct_temp` (optional WU outside refinement), `GIST_ID`/`GIST_TOKEN` (openings log) and `WU_*` secrets. No new secrets, no Telegram, no tado auth.

## Security

This repo is **public**. Anyone can read all code, workflow files, and run logs. Keep this in mind for every change.

- **Never add `pull_request` or `pull_request_target` triggers** to any workflow ‚Äî these allow forks to trigger runs and can expose secrets
- **Never log secrets** ‚Äî no `echo $SECRET`, no debug steps that dump env vars, no error handlers that print environment
- **GITHUB_TOKEN permissions must be minimal** ‚Äî workflows that only read code use `permissions: contents: read`; only workflows that commit/push use `contents: write`
- **No community actions without a pinned commit SHA** ‚Äî `uses: some-action@v2` is not safe; use `uses: some-action@<full-sha>` for any action outside the `actions/` namespace
- **No untrusted expression interpolation in `run:` steps** ‚Äî never put `${{ github.event.pull_request.title }}` or similar directly into shell commands; pass via `env:` and read from the environment instead
- **Secrets stay in GitHub Actions secrets** ‚Äî never hardcode credentials, never commit `.env` files
